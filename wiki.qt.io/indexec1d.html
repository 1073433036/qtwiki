Introduction

System Requirements

  • Git (>= 1.6.x)
  • Perl (>=5.16)
  • Python (>=2.7.x)
  • pscp from the PuTTY suite.
  • Windows 7 or Windows 8
  • Microsoft Visual Studio 2013 Express Edition or Full Version

Getting the sources

Qt 5

To build QtWebEngine on Windows, you will need a current build of the dev branch of Qt: 

git clone git://code.qt.io/qt/qt5.git qt5
cd qt5
git checkout dev

To save build time, it is possible to initialize the repository only with the Qt modules that are absolutely necessary: 

perl init-repository —module-subset=qtbase,qtxmlpatterns,qtdeclarative,qtquickcontrols,qtwinextras,qtwebengine

Installing additional dependencies

QtWebEngine depends on a couple of Microsoft SDKs for building. You can either install these manually one by one, or use the automatic toolchain setup from the chromium source tree.

Using the automatic setup

Clone Chromium's depot_tool repository http://www.chromium.org/developers/how-tos/install-depot-tools

From this repo, launch 

python win_toolchain\toolchain2013.py —targetdir <a > [—express]

This will download the necessary SDKs and extract them in the directory that was specified as targetdir. It will also generate a script called env.bat which will setup the necessary paths from the command line.

If you do not specify a directory, the toolchain will be stored in the subdirectory win_toolchain relative to the directory that the script was called from. It is recommended to not store the toolchain inside the repository, as e.g. a git clean would delete the SDKs.

Setting up dependencies manually:

Provided that you have already installed Visual Studio, you will need to install the following:

  • The Windows 8 SDK

When using the Express version of Visual Studio, you will also need to install:

  • Windows Driver Kit 7.1

After installation, you will need to customize the following template for a SetEnv.cmd file with the appropriate paths and copy it to the subdirectory of your installation of the Windows 8 SDK. (Note: You might need to run the copy command as Administrator if you installed the Windows SDK in Files 

@echo off
:: Targeting VS2012.
set PATH=<path-to-visual-studio>7\IDE;PATH
set INCLUDE=<path-to-win8sdk>;<path-to-win8sdk>;<path-to-visual-studio>if "%1"=="/x64" goto x64
set PATH=<path-to-win8sdk>86;<path-to-visual-studio>;<path-to-WDK>;PATH
set LIB=<path-to-visual-studio>;<path-to-win8sdk>8\um\x86
goto done
:x64
set PATH=<path-to-win8sdk>64;<path-to-visual-studio>86_amd64;<path-to-WDK>64;PATH
set LIB=<path-to-visual-studio>64;<path-to-win8sdk>8\um\x64
:done

Building the sources.

Automatic setup

If you used the automatic setup, change into the directory that the SDKs were installed to and run the environment variable setup batch script: 

cd <path to sdk installation>
env.bat

Manual setup

To easily setup the environment in a newly opened shell, it is recommended that you also make a batch file that can be run to set all the environment variables so you do not have to set them manually everytime you open a new shell: 

@echo off
set GYP_DEFINES=windows_sdk_path="<path-to-WDK>"
set GYP_MSVS_VERSION=2012e #or 2013e. For full version, remove the e
set GYP_MSVS_OVERRIDE_PATH=<path-to-visual-studio>
set GYP_GENERATORS=ninja
set GYP_PARALLEL=1
set WDK_DIR=<path-to-WDK>
set WindowsSDKDir=<path-to-win8sdk>
set PATH=<path-to-qt-bin-dir><path-to-qt-src>32\bin;<path-to-visual-studio>_NonRedist\x64\Microsoft.VC110.DebugCRT;<path-to-visual-studio>_NonRedist\x86
\Microsoft.VC110.DebugCRT;<path-to-visual-studio>64\Microsoft.VC110.CRT;<path-to-visual-studio>86\Microsoft.VC110.CRT;PATH

Qt

Once the repository is initialized, please configure your build as follows: 

configure -icu -nomake tests -nomake examples -opensource -confirm-license -debug -opengl es2 -angle

You may pass a prefix option to install Qt in a different directory or you may pass the -developer-build option. It is also possible to run a shadow build.

Make sure you have the bin directory of your Qt build and the gnuwin32/bin directory of your Qt repository in your PATH environment variable.

After configuration you can run the build as usual: 

nmake or jom
nmake install or jom install
